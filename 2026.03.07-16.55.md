## The Core Problem We Identified

The existing `install-openclaw.sh` is a good single-shot installer, but it has fundamental limitations as the infrastructure grows:

1. **Hardcoded to Anthropic** — provider is not configurable
2. **No concept of agent identity or role** — every install is identical
3. **No modular service installation** — Docker, Tailscale, Ansible, Ollama are all or nothing
4. **Not idempotent** — re-running on an existing install is risky
5. **Secrets handled insecurely** — API key passed as env var on the command line (visible in `ps`, shell history)
6. **No post-install verification** — only checks `gateway status`, not a real health checklist
7. **Not scalable to a fleet** — no mechanism for deploying multiple differentiated agents

## The Goals

### Immediate Goal
Replace `install-openclaw.sh` with a **modular, provider-agnostic, idempotent deploy system** that:
- Reads agent configuration from a YAML file in the repo
- Loads secrets from a file outside the repo (enforcing human-in-the-loop)
- Installs OpenClaw with the correct provider auth (Anthropic, OpenRouter, or Ollama)
- Optionally installs additional service modules (Docker, Tailscale, Ansible, Ollama)

## Key Design Decisions Made

### 1. Git Repo as Config Source of Truth
Rather than baking config into ocdevbase or hosting it at an HTTP endpoint, all agent configs live in the GitHub repo. On deploy, the VM clones the repo and reads its own config. This is:
- Immediately usable (no extra infrastructure needed)
- Versioned automatically
- Flexible (change a config, redeploy)
- Scalable (add a new `configs/ocdev02.yaml`, deploy a new agent)

### 2. Secrets Outside the Repo — Human-in-the-Loop by Design
API keys and auth tokens are **never** in the repo. They live at `~/.secrets/openclaw.sh` on each VM — a file the human must create manually before deployment can proceed.

The deploy script fails loudly if this file doesn't exist, with clear instructions on how to create it. This is intentional: it forces a moment of human intentionality before any credentials are used. Tony described this as "a man in the loop" — a human supervisor gate. It can be automated later once trust is established.

```bash
~/.secrets/openclaw.sh   ← lives here, chmod 600, never committed
```

The script also includes a paranoia check: it warns if the secrets file is accidentally inside a git repo.

### 3. Shell Scripts (for now), Ansible (for fleet management later)
The question of shell vs a more powerful tool (PyInfra was raised) was discussed carefully:

- **Shell** is the right tool for the bootstrap layer — it has zero dependencies and runs on a fresh VM before anything else is installed
- **Ansible** is the right tool for ongoing fleet management — Tony was already planning to install it, and it's designed exactly for managing multiple hosts consistently
- **PyInfra** is interesting but adds a third thing to learn; set aside for now

The deploy script installs Ansible as one of its modules, so by the time bootstrap is complete, the fleet management tool is ready.

The architectural boundary: shell gets you to "OpenClaw is running and Ansible is available." Everything after that is Ansible playbooks.

### 4. Module System
Each optional service (Docker, Tailscale, Ansible, Ollama) is a separate `lib/*.sh` file with a single `install_<name>()` function. Each module is:
- Idempotent (checks if already installed before acting)
- Independent (no inter-module dependencies)
- Optional (enabled/disabled via the agent's YAML config)

### 5. Provider-Agnostic Auth
OpenClaw handles different providers differently — a non-obvious gotcha discovered during testing:

| Provider | Auth mechanism | Model prefix required |
|----------|---------------|----------------------|
| Anthropic | `auth-profiles.json` with `type: "api_key"` | None (`anthropic/claude-...`) |
| OpenRouter | `openclaw.json` env var `OPENROUTER_API_KEY` | `openrouter/` prefix on model |
| Ollama | No auth needed | `ollama/` prefix on model |

The script handles all three cases and auto-prefixes model strings if the prefix is missing.

### Deploy Flow
```bash
# 1. Clone repo
git clone https://github.com/ajeless/ocdev.git && cd ocdev

# 2. Create secrets file (human must do this manually)
mkdir -p ~/.secrets && chmod 700 ~/.secrets
cp secrets.example.sh ~/.secrets/openclaw.sh
chmod 600 ~/.secrets/openclaw.sh
nano ~/.secrets/openclaw.sh   # fill in API keys

# 3. Deploy
bash deploy.sh ocdev01
```

- **Bug found:** OpenRouter provider threw `unknown model: anthropic/openrouter`

### Root Cause of the Bug
Two issues discovered by reading OpenClaw's local documentation:
1. OpenRouter model strings require the `openrouter/` prefix (e.g. `openrouter/google/gemini-2.0-flash`)
2. OpenRouter auth does not use `auth-profiles.json` — the API key must be set as `env.OPENROUTER_API_KEY` in `openclaw.json` via `openclaw config set env.OPENROUTER_API_KEY`

Both were fixed in a follow-up commit to the same branch. The script now auto-prefixes model strings and handles auth correctly for all three providers.

## Lessons Learned

- **OpenClaw provider auth is not uniform** — Anthropic and OpenRouter use completely different mechanisms. Always check the docs before assuming a pattern holds across providers.
- **Session logs are not ground truth** — prior session log said Docker/Ansible/Tailscale were installed; they weren't. Always verify actual system state.
- **The destroy/rebuild cycle is valuable** — it surfaces assumptions baked into the process and forces genuine reproducibility. Keep doing it.
- **Repo-scoped deploy keys are the right starting point** — easy to grant, easy to revoke, easy to audit. Expand to account-level SSH when the workflow is trusted.
- **Human-in-the-loop secrets gate is the right call early on** — automation that requires intentional human setup is safer automation, not weaker automation.
